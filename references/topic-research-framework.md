# 话题深度调研框架

## 调研原则

目标是为后续内容生产提供一份可靠的深度研究报告，不是堆搜索结果，也不是改编稿、制作稿或素材清单。

### 0. 默认联网

默认必须联网检索后再写。只有用户明确要求不联网、离线处理或只使用其提供的资料时，才跳过联网检索。

如果跳过联网检索，必须在「证据边界」写清楚材料来源和限制，不能把未核验信息写成确定事实。

### 1. 一手优先

按可信度排序：

1. 官方文档、论文、报告、公告、法律法规、原始数据。
2. 当事人访谈、演讲、公开长文、项目文档。
3. 权威媒体原创报道。
4. 行业分析、评论文章、百科和摘要类内容。

搜索引擎只能帮助发现来源，不能当作证据本身。

### 2. 从话题到研究问题

用户给的是话题，不一定是标题。要先把它改写成研究问题：

- 这个话题现在为什么值得研究。
- 读者最需要理解的核心矛盾是什么。
- 哪些事实已经确认，哪些仍在变化。
- 这件事对行业、市场、技术、政策或普通人有什么影响。

题目太宽时先自动收窄。不要直接问一串问题。

### 3. 事实、观点和推断分开

每条重要信息都标注类型：

- 事实：可核验的时间、数据、事件、定义。
- 观点：某个主体的解释、判断、主张。
- 推断：基于多条事实做出的综合判断。

推断必须写出依据，不能伪装成事实。

### 4. 当前状态校正

状态型话题要区分「原始来源当时怎么说」和「截至检索日期现在是什么状态」。常见状态型话题包括 IPO、上市、产品发布、监管生效、政策落地、投融资完成、价格变动、事故进展。

处理方式：

- 记录四个时间点：消息发布时间、原公告目标日期、检索日期、截至检索日期的状态。
- 如果原公告目标日期早于检索日期，必须继续查后续来源，不能只引用早期公告。
- 当前状态必须写成「已发生」「未发生」「截至检索日期未确认」或「公开资料有限」。
- 任何早于检索日期的状态事件，都不能和「预计」「预期」「估计」「计划」「expected」「scheduled」「slated」「will」这类前瞻词绑定。
- 最终交付物不要复述已经过期的前瞻口径。需要解释早期公告时，用「当时给出的目标日期」「当时的安排」这类中性说法。
- 找不到后续来源时，不要推断事件已经发生。
- 当前状态校正只服务于判断，不是报告正文。不要写「到某日期再看」「应写」「不能再写」「报告必须按」「合理表述是」「the report should」「the right framing is」。读者看到的正文应是直接事实、解释和影响。

### 5. 深度来自解释，不来自堆字数

报告至少要完成三层工作：

1. 事实层：发生了什么，谁说的，证据在哪里。
2. 机制层：为什么会这样，各因素如何互相影响。
3. 判断层：这件事最可能带来什么影响，哪些判断仍需要保留。

不要只列关键事实。每个核心事实后都要解释它为什么重要。

### 6. 保留争议

遇到冲突材料时，不要调和成一个顺滑结论。记录：

- 哪些来源互相冲突。
- 冲突发生在事实层、解释层，还是价值判断层。
- 当前更可信的一方及原因。
- 仍然无法确认的部分。

### 7. 面向后续再加工，但不替后续环节写稿

报告可以结构清楚，便于后续系统转换，但本 skill 不写：

- 页结构、页面标题、视觉建议。
- 口播稿、分镜、屏幕文字、制作节奏。
- 面向平台的标题、封面字、发布文案。

报告只交付研究本身。

## 最低来源标准

一般话题尽量满足：

- 总来源数：至少 8 个，深度话题尽量 10-15 个。
- 一手或权威来源：至少 3 个。
- 关键数据来源：每个核心数据都能回到原始出处。

如果达不到，仍可输出，但要在「证据边界」里说明，并降低结论确定性。

## 黑名单和谨慎来源

除非用户明确要求分析这些平台本身，否则不要把以下内容作为核心证据：

- 百科搬运、问答平台、无署名自媒体。
- 无法打开原文的二手转述。
- 没有日期、作者、出处的数据截图。
- 社交媒体单条爆料。

这些内容可以作为线索，但必须继续追到更可靠来源。
