# 话题深度调研报告

本 skill 只负责一件事：把用户给出的一个话题，调研成中英双语的深度研究报告。

不生成改编稿、制作稿或二次加工方案。后续系统基于本报告自行转换。

## 默认假设

如果用户只给一个话题，没有额外说明，按以下默认值执行：

- 输出语言：中英双语，分成两个文件。中文报告和英文报告信息一致，但不要逐句硬翻译。
- 输出目录：当前工作区下 `skill output/topic-research-report/<时间>-<话题>/`。
- 报告定位：完整深度调研报告，不是搜索摘要、改编素材或公众号文章。
- 写作风格：无 AI 味，只使用通用真人表达规则，不注入任何个人风格。
- 调研方式：默认必须联网检索后再写。除非用户明确要求不联网或只使用其提供的资料，不能只凭模型记忆生成。
- 受众：对话题有兴趣但不是专家的普通内容创作者或知识型读者。

默认直接执行。只有当话题范围会明显影响结论时才追问。例如用户只说「AI」过于宽泛，先自动收窄成一个可研究的问题；仍无法合理收窄时再问用户。不要把流程变成问卷。

## 快速启动

先创建输出包：

```bash
python3 scripts/create_topic_package.py "用户输入的话题" --base "skill output/topic-research-report"
```

脚本会生成：

```text
<输出目录>/
├── 调研报告-中文.md
└── research-report-en.md
```

随后完成调研、填充两份文件，并运行：

```bash
python3 scripts/check_topic_outputs.py "<输出目录>"
```

## 工作流

### Step 1: 把话题改写成研究问题

把用户输入改写成一个可调研的问题：

- 主题对象：要研究什么。
- 时间范围：历史全貌、最近进展，还是某个事件窗口。
- 研究主线：这份报告最终要解释什么。
- 受众边界：普通读者需要哪些背景，哪些专业细节可以压缩。

边界用于内部判断，不要写成报告开头的表单。报告应是标题后直接进入正文；检索日期、来源限制和研究边界放在文末「证据边界 / Evidence Limits」。

只在以下情况暂停问用户：

- 话题同时包含多个互斥方向，无法判断主线。
- 话题涉及高风险建议，如医疗、法律、投资决策，需要确认用途和免责声明边界。
- 用户明确要求只用指定资料，但没有提供资料。

除此之外，按默认假设继续执行。

### Step 2: 调研

先读取 `references/topic-research-framework.md`。

默认先联网检索。用户提供的本地资料可以作为优先来源，但只要没有明确要求「只用本地资料」，就继续用联网检索补齐事实、时间线、数据和争议。

最低要求：

- 可行时至少收集 8 个有效来源，深度话题尽量达到 10-15 个。
- 至少 3 个一手或权威来源；如果达不到，在「证据边界」写明。
- 每个关键判断都要能追到来源，不写无来源断言。
- 区分事实、观点、推断。
- 保留争议和不确定性，不把材料硬凑成单一结论。
- 涉及时效性内容时写明检索日期。

对 IPO、上市、产品发布、监管生效、投融资、价格变动、政策节点这类状态型话题，必须做当前状态核对：

- 先列清楚消息发布时间、原公告目标日期、检索日期、截至检索日期的状态。
- 如果原公告目标日期早于检索日期，必须继续查后续来源，不能停留在早期公告口径。
- 任何早于检索日期的状态事件，都不能和「预计」「预期」「估计」「计划」「expected」「scheduled」「slated」「will」这类前瞻词绑定。
- 写当前状态时只能写「已发生」「未发生」「截至检索日期未确认」或「公开资料仍不足」。
- 最终交付物不要为了举反例而复述过期前瞻词。需要解释早期公告时，改写成：「6 月 11 日公告给出 6 月 12 日交易安排；截至 6 月 22 日，后续来源显示已开始交易。」
- 当前状态核对是写作前的内部校验，不要把校验过程写给读者看。正文不能出现「到某日期再看」「应写」「不能再写」「报告必须按」「合理表述是」「the report should」「the right framing is」这类质检腔。正文只写自然事实和判断，例如：「SpaceX 6 月 11 日完成定价，6 月 12 日开始交易。」

### Step 3: 写作前去 AI 味

写报告前，先读取 `references/human-writing-rules.md`。

硬规则：

- 不使用中文或英文破折号。
- 不编数字、日期、来源、人物、机构或引用。缺证据就标注「待核实」或 `(to verify)`。
- 不写客服腔、总结腔、宣传腔。
- 不用「首先/其次/此外/综上所述」这类机械路标堆结构。
- 不用虚假施动，比如「数据告诉我们」「市场奖励」「技术正在重塑」。
- 英文用普通、直接、清楚的词，不把 plain English 升级成 fancy English。
- 中英文表达要信息一致，但不要逐句硬翻译。中文像中文，英文像自然英文。

### Step 4: 写深度调研报告

按 `references/report-template.md` 填写 `调研报告-中文.md` 和 `research-report-en.md`。

报告必须像研究报告，而不是材料清单或填空表。它要回答：

- 这个话题到底是什么。
- 这件事为什么重要。
- 已核实的关键事实是什么。
- 事件或机制如何演变。
- 主流解释、反方解释、争议点分别是什么。
- 哪些判断能成立，哪些只能作为假设。
- 后续最该观察什么。

写法要求：

- 标题后直接进入正文，不输出「选题定义」「研究结论」「执行摘要」这类元信息开头。
- 先给完整结论，再展开证据。
- 背景、事实、分析、争议、影响、证据边界分清楚。
- 每个核心观点都要有证据、解释和限制条件。
- 不写成分镜、页结构、屏幕文字、口播稿或其他二次加工稿。
- 中文文件只写中文，英文文件只写 English。

### Step 5: 失败模式与兜底

| 触发条件 | 一线处理 | 仍失败时 |
|---|---|---|
| 话题太宽 | 自动收窄到一个可研究问题，并在开头正文自然交代范围 | 暂停问用户选方向 |
| 可靠来源少于 3 个 | 继续产出，但在「证据边界」写明公开资料有限 | 把结论降级为「可讲假设」，不写成定论 |
| 找到的资料互相冲突 | 在「争议与不确定性」并列呈现 | 不强行调和，标注无法确认 |
| 原公告目标日期早于检索日期 | 继续查后续来源，更新成当前状态 | 查不到后续来源时写「截至检索日期未确认」，不要继续写前瞻口径 |
| 报告太像素材包 | 增加背景、机制、解释、争议、影响、后续观察 | 删除二次加工导向内容 |
| 中英文信息不一致 | 以来源证据为准重写其中一版 | 不能确认时标注待核实 |
| 校验脚本失败 | 按错误项逐条修复 | 客观无法修复时在最终回复说明原因 |

### Step 6: 校验

运行校验脚本：

```bash
python3 scripts/check_topic_outputs.py "<输出目录>"
```

修复所有失败项后再交付。若某项因为客观原因无法满足，例如公开来源少，在最终回复里明确说明。

校验失败时不要只改措辞。先回到来源核对事实状态，再改报告。

## 不要做什么

- 不要输出口播稿、分镜脚本、页结构或其他制作稿。
- 不要把调研报告写成搜索摘要、公众号文章或素材清单。
- 不要把报告开头写成「用户原始话题 / 研究问题 / 目标读者 / 检索日期」表单。
- 不要编造来源、数字、人物、机构或引文。
- 不要引用本机路径、其他本地 skill 名称或私有资料来源。
- 不要加入任何个人风格、口癖或人设。
- 不要用 AI 味套话填充篇幅。
- 不要在来源不足时写确定性结论。

## 交付格式

最终回复只需要给用户：

- 中文调研报告路径。
- English research report path.
- 校验结果。
- 重要限制，例如来源不足或话题被合理收窄。

不要把两份长文完整粘贴到聊天里，除非用户明确要求。
